# Executive Rooms Accessibility and UX Gate

Status: required before runtime approval.

## Executive experience

The room purpose, membership visibility, retention period, moderator identity, and Atlas state must be understandable before an executive posts. The interface must never resemble a public feed or create pressure through engagement counts.

## Accessibility acceptance

- All room actions are keyboard reachable in a predictable order.
- Focus is visible and returns to the initiating control after dialogs close.
- New-message announcements use a polite live region and never interrupt reading.
- Author, timestamp, reply relationship, moderation state, and Atlas provenance have screen-reader labels.
- Colour is never the only indicator of role, confidence, warning, or moderation state.
- Reduced-motion preferences disable nonessential animation.
- Zoom, text reflow, and 390px layouts preserve reading and posting without horizontal scrolling.
- Destructive and audience-changing actions require explicit, labelled confirmation.
- Loading, empty, unavailable, archived, and access-denied states explain the next safe action.

## Cognitive-load constraints

The initial proof exposes only room topic, chronological discussion, reply, pin, private bookmark, member list, and explicit Atlas request. Presence is off by default. Notification controls are quiet by default. No streaks, popularity rankings, reaction totals, or infinite feed are permitted.

## Validation evidence required

Keyboard-only journey, screen-reader smoke test, contrast review, 200% zoom, reduced motion, desktop, and 390px mobile must pass before Founder runtime approval.
