# Founder Acceptance

> Purpose: Preserve founder-observed evidence for Release 0.6 RC1 without treating automated tests as product acceptance.

This folder records the founder journey, issue register, executive-experience assessment, and release decision. The founder acts as the first Executive Design Partner using fictional information only. Credentials, invitation secrets, verification links, tokens, and unnecessary personal information must never be recorded.

The observer updates the records immediately after each stage. A claim is marked `Passed`, `Failed`, `Blocked`, or `Not Tested`. Empty evidence is never converted into a pass.

Files:

- `FOUNDER_ACCEPTANCE_REPORT.md` — chronological journey evidence.
- `FOUNDER_ACCEPTANCE_ISSUES.md` — reproducible defects and product concerns.
- `FOUNDER_EXPERIENCE_REVIEW.md` — executive-product evaluation.
- `RELEASE_DECISION.md` — the evidence-based release recommendation.
- `screenshots/` — sanitized visual evidence; secrets and credentials must never appear.

Current environment: local RC1 validation. No staging, production, real personal data, or design-partner invitation is authorized by these records.
