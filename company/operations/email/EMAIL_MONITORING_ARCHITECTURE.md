# Email Monitoring Architecture

Purpose: Define a secure, narrow, and reviewable design for detecting Microsoft responses concerning the Orendalis DKIM support case without granting uncontrolled mailbox authority.

## Current Readiness

Microsoft support case `2607130050001139` is open. The monitoring design is ready for controlled configuration, but no mailbox connection, flow, test, or live monitoring has been activated.

## Architecture

```text
Founder mailbox
    ↓ new-message trigger
Power Automate / Office 365 Outlook connector
    ↓ narrow relevance and authenticity checks
Case-matched branch ──→ safe alert email to founder-approved external address
    ├─────────────────→ apply "Microsoft Support — DKIM" category
    └─────────────────→ record message identifier and flow outcome for deduplication

Suspicious branch ───→ flag for manual inspection; no links or attachments opened
Unmatched branch ────→ no action and no content retention
```

## Matching Rule

A message is relevant only when multiple independent signals support the match. Sender display name alone is never sufficient.

### Required case signal

At least one must be true:

- Subject contains case number `2607130050001139`.
- Body contains case number `2607130050001139`.
- Microsoft supplies a verified case/thread identifier that maps to the same case.

### Supporting topic signal

At least one must be present in the subject or a minimal text inspection:

- DKIM
- `CnameMissing`
- Exchange Online
- selector publication

### Sender and authentication signal

- Use the actual sender address and domain, not the display name.
- Require the domain to match a reviewed allowlist derived from a genuine Microsoft case message or Microsoft support documentation.
- Where headers are available, require satisfactory SPF, DKIM, and DMARC results or route the message to suspicious review.
- Treat look-alike domains, reply-to mismatches, failed authentication, unexpected attachments, and shortened links as suspicious.

The case number plus topic signals may identify relevance, but they do not prove authenticity. Any message that fails sender or authentication review must not be trusted automatically.

## Data Minimization

The flow may capture only:

- Received date and time
- Sender address
- Subject
- Case number
- Short, non-confidential status summary
- Whether founder action appears required
- Safe Outlook message link, when generated by Microsoft and supported
- Stable message identifier used for duplicate prevention

It must not copy attachments, full message bodies, credentials, tokens, recovery information, or unrelated mailbox content into alerts or logs.

## Duplicate Prevention

Use the Outlook internet message identifier, or the connector's immutable message identifier when the former is unavailable, as an idempotency key. Before sending an alert, check a restricted record of processed identifiers. Record the identifier only after a successful alert. A retry must update the existing audit entry rather than send a second alert.

The persistent store must remain inside the Orendalis Microsoft tenant and expose no message body. Its retention period should be reviewed before activation.

## Failure Handling

- **Connector unavailable:** Record the failed run and alert through Power Automate's owner notification; do not broaden permissions.
- **Alert delivery failure:** Retry within bounded limits, then surface a failed run for manual review.
- **Cannot determine authenticity:** Apply the suspicious-review path and do not open links or attachments.
- **Cannot summarize safely:** Send metadata only and instruct the founder to open Outlook.
- **Category action fails:** Preserve the notification result and report the category failure separately.
- **Flow disabled or expired connection:** Reauthenticate interactively; never store credentials in documentation.

## Recovery and Audit

The flow owner should review run history after activation, monthly thereafter, and after any failure. Audit records should show trigger time, match outcome, alert outcome, category outcome, deduplication key, and error state without retaining private content. Export the flow definition after successful testing and after approved changes; keep connection secrets outside the repository.

## Security Boundaries

- Scope the Office 365 Outlook connection to the founder mailbox only.
- Do not use tenant-wide application permissions.
- Do not mark messages read.
- Do not automatically forward original messages or attachments.
- Do not click links or open attachments.
- Do not send replies.
- Do not activate the flow until the controlled test passes and the founder approves live activation.

## Known Limitations

- Power Automate polling can be delayed, and Microsoft documents rare trigger delays of up to one hour.
- Connector retries can create duplicate actions unless explicit deduplication is implemented.
- Protected messages may expose limited trigger content.
- Sender-domain allowlisting alone cannot establish authenticity.
- Licensing entitlements must be verified in the Orendalis tenant before configuration.

## Approval Gates

Separate founder approval is required before connecting Power Automate to Outlook and again before activating the tested flow for live monitoring.
